# Ship Studio — End to End Product Spec

> Configuration studio for AI agents. CLI + web app + registry.
> One place to compose, manage, and distribute agent configurations.

---

## Vision

Ship Studio is the professional environment where developers craft how their AI agents work. Not a settings page. Not a dotfiles manager. A studio — compose modes from skills and MCP servers, compile to any provider, hotswap from the CLI, share with a URL.

The web app is free and gets people in the door. The CLI is free and gets `.ship/` into repos. The registry is the community. Pro is for teams and power users who need more.

---

## Core Concepts

### Mode
A named agent configuration. References skills, MCP servers, permissions, rules, and provider targets. Lightweight — stores references not content. Compiles to provider-native formats on demand.

### Skill
An Agent Skills spec-compliant package. SKILL.md + optional scripts/, references/, assets/. Versioned. Cached locally. Loaded into agent context when the mode activates.

### MCP Server
A Model Context Protocol server configuration. Transport type (stdio/http/sse), connection details, tool-level allow/deny filter, env var requirements.

### Path Context
A mode bound to a filesystem path. Different paths can have different active modes. The foundation of the workspace concept — each project directory has its own agent configuration.

### Compiled Output
The provider-native files Ship generates from a mode: CLAUDE.md + .mcp.json for Claude Code, AGENTS.md + .codex/config.toml for Codex, GEMINI.md + .gemini/settings.json for Gemini CLI, .cursorrules for Cursor.

## File System Layout

### Global (~/.ship/)
Personal configuration. Never committed to git.

```
~/.ship/
├── config.toml           # identity, defaults, API keys (encrypted)
├── ship.key              # local server auth token
├── modes/                # personal saved modes
│   ├── rust-expert.toml
│   └── frontend-react.toml
├── skills/               # installed skills (Agent Skills spec)
│   ├── rust-expert/
│   │   ├── SKILL.md
│   │   └── references/
│   └── @anthropic/
│       └── memory/
├── mcp/                  # MCP server configs
│   ├── github.toml
│   └── rust-docs.toml
└── cache/                # registry fetches, 24h TTL
    ├── registry-mcp.json
    └── registry-skills.json
```

### Project (.ship/)
Per-project configuration. Committed to git (secrets excluded via .gitignore).

```
.ship/
├── ship.toml             # project config, provider targets, mode ref
├── .gitignore            # excludes .ship/secrets/, compiled artifacts
├── modes/                # project-specific modes (override global)
├── agents/
│   ├── skills/           # project-local skills
│   └── rules/            # always-on project rules
└── secrets/              # gitignored, env vars, tokens
```

### Compiled Artifacts
Generated by Ship. Never committed. Gitignored by default.

```
project/
├── CLAUDE.md             # compiled for Claude Code
├── .mcp.json             # MCP config for Claude Code
├── AGENTS.md             # compiled for Codex
├── .codex/config.toml    # Codex config
├── GEMINI.md             # compiled for Gemini CLI
└── .gemini/
    └── settings.json
```

**On filesystem overwrites:**
Yes — Ship owns these files when Ship is managing the project. Running `ship compile` overwrites existing provider config files. This is intentional and documented clearly. The recommendation: commit `.ship/`, gitignore the compiled artifacts, let Ship regenerate them. For projects with existing Claude Code configs, `ship import` reads the existing files and converts them to Ship mode format before overwriting.

## Mode Format

```toml
# ~/.ship/modes/rust-expert.toml
[mode]
name = "Rust Expert"
id = "rust-expert"
version = "1.0.0"
description = "Systems programming with safety-first refactors"
providers = ["claude", "codex"]

[skills]
refs = [
  "ship:rust-expert@1.2.0",
  "ship:test-driven@latest",
  "@anthropic/memory@2.0.0"
]

[mcp]
refs = ["ship:rust-docs", "ship:cargo-tools"]
servers = [
  { id = "github", tools_allow = ["mcp__github__*"], tools_deny = [] }
]

[permissions]
preset = "ship-guarded"
tools_deny = ["mcp__*__delete*", "mcp__*__exec*"]
fs_allow = ["**/*"]
fs_deny = ["/etc/**", "/sys/**", "~/.ssh/**"]

[rules]
inline = """
Keep operations deterministic.
Prefer explicit failures over silent fallback.
Run tests before marking work done.
"""

[context]
always_include = ["README.md", "ARCHITECTURE.md"]
ignore = [".ship/secrets/**", "node_modules/**"]
```

## CLI

### Installation
```bash
curl -fsSL https://getship.dev/install | sh
# or
cargo install ship-cli
# or  
brew install ship
```

### Authentication
No login required to use core features. Lazy auth — prompts only when a cloud feature is attempted.

```bash
ship login           # authenticate with getship.dev
ship logout
ship whoami
```

### Init
```bash
ship init                    # scaffold .ship/ in current directory
ship init --global           # configure ~/.ship/ identity and defaults
ship init --provider claude  # init with specific provider target
```

### Mode Management
```bash
# Use a mode
ship use rust-expert                    # activate local mode
ship use @anthropic/frontend-react      # install + activate registry mode
ship use https://getship.dev/m/abc123   # install + activate from URL

# Path-aware mode switching
ship use rust-expert --path ~/projects/myapp
ship use frontend-react --path .        # current directory
ship status                             # show active mode per path

# List modes
ship modes                              # all available modes
ship modes --local                      # local ~/.ship/modes/ only
ship modes --project                    # .ship/modes/ only
ship modes --cloud                      # cloud saved modes

# Manage modes
ship mode create my-mode                # create new mode interactively
ship mode edit my-mode                  # open in Studio (launches web app)
ship mode edit my-mode --editor code    # open in VS Code
ship mode delete my-mode
ship mode clone rust-expert my-rust     # clone and customize
ship mode publish my-mode               # publish to marketplace (account required)
```

### Compilation
```bash
ship compile                    # compile active mode to all provider targets
ship compile --provider claude  # compile for specific provider only
ship compile --dry-run          # preview output without writing files
ship compile --watch            # recompile on mode changes
```

### Skills
```bash
ship skill add rust-expert                    # install from Ship registry
ship skill add @anthropic/memory              # install from registry with namespace
ship skill add ./local-skill                  # install from local path
ship skill add https://github.com/org/skill   # install from GitHub
ship skill list                               # list installed skills
ship skill remove rust-expert
ship skill update                             # update all skills
ship skill create my-skill                    # scaffold new skill (Agent Skills spec)
ship skill validate ./my-skill                # validate against spec
ship skill publish ./my-skill                 # publish to registry
```

### MCP Servers
```bash
ship mcp add github                     # add from registry
ship mcp add --url http://localhost:7773 # add custom HTTP server
ship mcp list                           # list configured servers
ship mcp probe                          # test all server connections
ship mcp probe github                   # test specific server
ship mcp remove github
```

### Import / Export
```bash
ship import                      # detect + import existing provider configs
ship import --path ~/projects/x  # import from specific path
ship export claude               # export current mode to Claude Code format
ship export all                  # export to all configured providers
ship export --zip                # download all formats as zip
```

### Sync
```bash
ship sync                        # sync local modes to cloud
ship sync --pull                 # pull cloud modes to local
ship sync --push                 # push local modes to cloud
```

### Local Server
The CLI runs a local HTTP server on port 7701. Used by the web Studio to access local features — MCP probing, shell autocomplete, installed provider detection.

```bash
ship server start    # start local server (auto-starts with ship use)
ship server stop
ship server status
```

### Desktop Integration
```bash
ship mode edit my-mode    # opens Studio in browser at /studio/modes/my-mode
                          # when Ship desktop app is installed, opens desktop app instead
                          # this is the bridge between CLI and visual editing
```

## Path-Aware Mode System

Each filesystem path can have an active mode. This is the foundation of the workspace concept.

```bash
~/projects/
├── rust-api/        → active mode: rust-expert
├── react-dashboard/ → active mode: frontend-react  
└── docs-site/       → active mode: documentation
```

```bash
# Set mode for a path
ship use rust-expert --path ~/projects/rust-api

# Ship detects path on cd (shell hook)
cd ~/projects/rust-api
ship status
# → rust-expert (rust-api) — claude, codex

# Auto-compile on path switch (optional, shell hook)
# Rewrites CLAUDE.md etc. in the project directory
```

Shell hook (added to .zshrc/.bashrc by ship init):
```bash
function _ship_cd() {
  cd "$@" && ship compile --silent --if-changed
}
alias cd=_ship_cd
```

This is the first step toward workspaces. Path → mode → compiled config. The project directory always has the right agent configuration for the work being done there.

## Registry

### MCP Registry
Proxies the official MCP registry at registry.modelcontextprotocol.io. No maintenance burden — Ship fetches and caches the manifest.

```
GET https://getship.dev/api/mcp          # all servers
GET https://getship.dev/api/mcp/:id      # specific server
GET https://getship.dev/api/mcp/search?q # search
```

### Skills Registry
Ship-curated library seeded with:
- Anthropic official skills
- Vercel-labs top skills (from skills.sh)
- Ship workflow skills (ship-workflow, skill-creator, etc.)
- Community submissions (manually reviewed, Ship-verified badge)

```
GET https://getship.dev/api/skills           # all skills
GET https://getship.dev/api/skills/:id       # specific skill
GET https://getship.dev/api/skills/:id@:ver  # versioned
GET https://getship.dev/api/skills/search?q  # search
```

Skills are stored in Ship's GitHub registry repo. Community submissions via PR. Ship-verified badge requires manual review. Versioned via git tags.

### Mode Registry (Marketplace)
Community-published modes. Account required to publish.

```
GET  https://getship.dev/api/modes         # browse
GET  https://getship.dev/api/modes/:id     # specific mode
POST https://getship.dev/api/modes         # publish (auth required)
```

### URL Import (No Account)
```
https://getship.dev/m/:id     # temporary mode URL, 90-day TTL
https://getship.dev/m/@user/mode-name  # permanent (account holder)
```

Anyone can generate a shareable mode URL from Studio without an account. The URL resolves to the mode TOML. Install with `ship use <url>`.

## Desktop App Integration

The CLI is the bridge between Studio and the desktop app.

```bash
ship mode edit my-mode
```

- No desktop app: opens browser at getship.dev/studio?mode=my-mode
- Desktop app installed: opens Ship desktop app with that mode loaded in Studio view
- Desktop app is Ship full — project management + Studio + sessions

This means Studio in the browser and Studio in the desktop app are the same surface with the same component library (packages/ship-ui). The desktop app adds local context and the full project management layer. The browser app is the entry point.

The desktop app is Ship v1 for project management. It ships when the workflow is proven. Studio ships first.

## Distribution Strategy

`.ship/` in repos is the growth engine.

```
ship init in any project
  → .ship/ship.toml committed to git
    → teammates see it, ask what it is
      → README in .ship/ links to getship.dev
        → they visit Studio, build their own mode
          → cycle repeats
```

The README auto-generated in `.ship/`:
```markdown
# Ship Configuration

This project uses [Ship](https://getship.dev) to manage AI agent configuration.

Install the CLI: `curl -fsSL https://getship.dev/install | sh`
Then: `ship use` to activate the project mode.
```

Every `.ship/` directory is a planted flag. Every developer who clones the repo is a potential Ship user.

## What We Don't Build (Yet)

- Our own skill runtime / execution environment (use Agent Skills spec, defer VM sandbox to Pro v2)
- Our own MCP registry (proxy the official one)
- Signing infrastructure (watch TanStack Intent, integrate when it wins)
- Native IDE extensions (CLI + web covers the use case, extensions are Phase 3)
- Admin / managed config (Pro roadmap, after team configs proven)
- Mobile app (web is responsive, CLI is terminal)
