# Shipwright Alpha (v0.1.0)

## The Goal
The Alpha release focuses on delivering the **Core Loop** works end-to-end without friction: 
`Note → Spec → Feature → Issues → Agent Session → Merge`

## Relation to Vision
Shipwright is the **"Structure Layer your AI agents don't have."** This release grounds that layer by ensuring project memory is:
1. **Persistent**: Stored as version-controlled Markdown in `.ship/`.
2. **Context-Aware**: Injected automatically into agent sessions on `git checkout`.
3. **Actionable**: Accessible via standard MCP tools that agents are native to.

## Alpha Core Pillars

### 1. Unified Project Shell
The infrastructure that makes a directory "Ship-aware." Includes zero-config initialization (`ship init`), Git hook installation, and automated `.gitignore` management to keep generated state out of the repository.

### 2. Canonical Documentation Primitives
Standardized, TOML-frontmattered Markdown schemas for **Issues**, **Specs**, **ADRs**, and **Notes**. These ensure that both humans and agents have a predictable, structured high-fidelity memory of the project.

### 3. Automated Context Sync
The "Bridge" between the repo and the agent. A Git hook engine that detects branch context on checkout and instantly generates `CLAUDE.md`, `.mcp.json`, and `.gemini/settings.json` tailored to the active Spec.

### 4. Shipwright MCP Tool Surface
Exposes the project runtime to any MCP-compatible agent (Claude Code, Cursor, Windsurf). Agents gain the ability to list specs, create issues, and refine ADRs directly through their tool calls.

### 5. Workflow Lifecycle Engine
The state machine for project work. Features include promoters (Note → Spec), automated branch-to-feature linking via SQLite, and "File-as-Status" folder semantics for tactile project management.

---

## Validation Goal
**Alpha Dogfooding**: Validating the full loop by using Shipwright to build Shipwright. Every friction point found during current development informs the Alpha quality bar.
