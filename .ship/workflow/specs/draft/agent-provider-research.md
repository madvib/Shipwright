### Deep Dive: The 4 Paradigms of Permissions & Enforcement

As you noted, Permissions are the most highly fragmented aspect of the agent ecosystem. To build your `UnifiedAgentConfig` schema in Rust, you must account for 4 distinct paradigms currently in use:

**1. Deterministic Event Hooks (The Claude Code / OpenCode Approach)**
This is the most secure paradigm. Providers like **Claude Code** and **OpenCode** do not rely on the LLM to follow rules. Instead, they trigger scripts during the execution lifecycle.

- **Claude Code:** Exposes hooks like `PreToolUse`, `PostToolUse`, and `Stop` in `~/.claude/settings.json`. If an LLM attempts to read an `.env` file via a filesystem MCP, a Python/Shell script intercepts the `tool_name` and `tool_input` via JSON stdin. Returning exit code `2` deterministically blocks the operation.
- **OpenCode:** Uses TypeScript/JavaScript plugins that hook into `permission.asked` or `tool.execute.before` to reject actions.

**2. Granular Tool Filtering (The Gemini Code Assist / Roo Code Approach)**
This paradigm filters the _availability_ of specific commands or tools before the LLM can even see them.

- **Gemini Code Assist:** Uses explicit arrays in `~/.gemini/settings.json`. You can allowlist specific commands using `"coreTools": ["ShellTool(ls -l)"]` or blocklist using `"excludeTools": ["ShellTool(rm -rf)"]`.
- **Roo Code:** Uses `.rooignore` to hard-block the `read_file`, `write_to_file`, and `apply_diff` tools from accessing restricted paths. It also uses UI toggles to blanket-approve/deny access to high-risk categories (like terminal execution or file editing).

**3. State/Mode Restrictions (The Void Editor / Zed Approach)**
Security is bounded by the profile or "mode" the agent is spawned in.

- **Void Editor:** Has a "Gather Mode" which acts as a restricted, read-only version of their Agent mode. It literally disables the execution of any modification tool at the architectural level.
- **Zed:** Allows the creation of dedicated agent profiles in `settings.json` that lock an agent strictly to container-only tools, isolating it entirely from the host filesystem.

**4. Policy Enforcement (The Cursor Approach)**

- **Cursor:** Focuses on organizational governance. "Team Rules" are managed from a centralized dashboard and can be set to "Enforce this rule," which completely prevents local developers from toggling the rule off or overriding it via their local `.cursor/rules`.
