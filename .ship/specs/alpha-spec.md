# Shipwright — Alpha Specification

**Version:** 0.2-alpha  
**Status:** Active  
**Last Updated:** 2026-02-25

---

## The Alpha in One Sentence

A local-first project OS: markdown files in git, a clean UI, an MCP server that persists project memory across sessions, git hooks that auto-configure your AI agent on every branch checkout, and just enough structure to go from idea to working issues.

---

## The Core Loop

```
Note → Spec → Extract Issues → Work Issues (human or agent) → Update Issues → Repeat
```

Everything in the alpha exists to serve this loop. If a feature doesn't directly support it, it doesn't ship in alpha.

---

## What Alpha Is Not

- No external MCP management UI (config export CLI only)
- No cloud sync
- No user accounts or auth
- No cross-project linking
- No custom document types
- No agent session spawning or orchestration
- No marketplace
- No mobile
- No component library extraction (`@shipwright/ui` — V1)
- No third-party extensions (TypeScript/V8 — V2)
- No parallel worktree orchestration (V2)

| File                       | Format                                            |

**Status is the folder.** Moving an issue = moving a file. No status field in frontmatter.

**Generated files are never committed.** CLAUDE.md, .mcp.json, and all AI CLI configs are written by Shipwright on branch checkout and gitignored. Manually maintaining these files in a Shipwright project is an error — they will be overwritten and may contain context-specific content that corrupts other branches.

## SQLite Runtime Layer

Shipwright uses two SQLite databases. All access is through typed Rust repositories — no raw SQL outside `crates/runtime/src/db/`.

**`~/.ship/shipwright.db`** — global state:

- Registered projects
- Global mode state
- Global MCP connection state

**`.ship/ship.db`** — project state (gitignored):

- Worktree registry and status
- Agent session records
- Orchestration locks
- Branch context cache (spec linkage, open issues, generated file hashes)
- Task queue (alpha stub — used in V2)
- Mode override state

Migrations run automatically via `sqlx::migrate!()` on startup. Zero user interaction.

The discipline: if a human would ever read, diff, or commit it — markdown. If it's ephemeral, concurrent, or machine-generated churn — SQLite.

## Crate Structure

```
shipwright/
├── crates/
│   ├── runtime/          # Core — document store, MCP, git hooks, SQLite, event bus
│   │   └── src/
│   │       ├── db/       # SQLite repositories (all db access lives here)
│   │       ├── document/ # Generic document model + file store
│   │       ├── mcp/      # MCP server + dynamic tool registry
│   │       ├── events/   # Async event bus
│   │       ├── config/   # Project + global config
│   │       ├── log/      # Action log writer
│   │       └── store/    # Per-module scoped key-value
│   │
│   ├── sdk/              # ShipwrightModule trait + ModuleContext
│   │
│   ├── cli/              # CLI — thin layer over runtime
│   │
│   ├── ui/               # Tauri app shell
│   │   ├── src/          # React + TypeScript
│   │   └── src-tauri/    # Tauri 2.x commands (specta-typed)
│   │
│   └── modules/          # First-party modules
│       ├── issues/       # Kanban, issue document type, MCP tools
│       ├── specs/        # Split editor, spec document type, MCP tools
│       ├── adrs/         # ADR list, adr document type, MCP tools
│       ├── notes/        # Notes list, note document type, MCP tools
│       └── git/          # Hooks, worktrees, context generation
│
└── Cargo.toml            # Workspace
```

**Why `runtime/` not `logic/`:** The runtime is the OS substrate — document store, MCP server, event bus, SQLite layer. "Logic" undersells it and misrepresents what it does. Renaming now costs a morning. Renaming at 90K LOC costs a week.

**Why `modules/` not `plugins/`:** Modules are compiled-in, full-trust, first-party code. "Plugin" implies third-party, sandboxed, optional. These aren't plugins — they're the application. Third-party TypeScript extensions come in V2 and will be called extensions.

## Module Trait

```rust
// crates/sdk/src/lib.rs

pub trait ShipwrightModule: Send + Sync {
    fn manifest(&self) -> &ModuleManifest;
    fn on_load(&self, ctx: &ModuleContext) -> Result<()> { Ok(()) }
    fn on_unload(&self, ctx: &ModuleContext) -> Result<()> { Ok(()) }
    fn document_types(&self) -> Vec<DocumentTypeSpec> { vec![] }
    fn mcp_tools(&self) -> Vec<McpToolSpec> { vec![] }
    fn cli_commands(&self) -> Vec<CliCommandSpec> { vec![] }
    fn ui_contributions(&self) -> UiContributions { UiContributions::default() }
    fn on_document_created(&self, ctx: &ModuleContext, doc: &Document) -> Result<()> { Ok(()) }
    fn on_document_updated(&self, ctx: &ModuleContext, doc: &Document) -> Result<()> { Ok(()) }
    fn on_document_moved(&self, ctx: &ModuleContext, doc: &Document, from: &str, to: &str) -> Result<()> { Ok(()) }
}
```

The test: remove every module. Shipwright should still boot, show an empty canvas, and be fully functional — just with nothing registered.

## Git Module

The git module is alpha scope. It's the feature that turns Shipwright from a project management tool into an agent infrastructure layer.

### Hooks

`ship init` installs hooks into `.git/hooks/<hook>.d/shipwright` (not directly into `.git/hooks/<hook>`) so Shipwright coexists with husky, lefthook, Claude Code, Codex, and any other hook managers.

```
post-checkout     → generate CLAUDE.md + .mcp.json for current branch
pre-commit        → enforce gitignore rules (block staging of generated files)
prepare-commit-msg → inject active issue context into commit message
post-merge        → archive completed issues, update branch status in SQLite
```

All hooks are configurable per project:

```toml
# .ship/config.toml

[git.hooks]
post_checkout.generate_context = true
pre_commit.enforce_gitignore = true
pre_commit.validate_consistency = true
prepare_commit_msg.inject_issue = true
post_merge.archive_completed = true
```

### Branch-Scoped Context Generation

When `post-checkout` fires, Shipwright:

1. Reads the current branch
2. Checks SQLite branch context cache (skips if unchanged)
3. Finds the linked spec (by naming convention or explicit config)
4. Reads open issues, relevant ADRs
5. Generates `CLAUDE.md` from spec + issues + ADRs + active skills
6. Generates `.mcp.json` for managed AI CLIs (alpha: Claude Code only)
7. Updates SQLite branch context cache

The `shipwright-workflow.md` skill is always included in every CLAUDE.md. It teaches the agent how to use Shipwright's MCP tools — when to create issues, when to file ADRs, how to move issues through status. This is what makes agents "Shipwright-native" without any user configuration.

### Branch-Spec Linking

```toml
# .ship/config.toml — naming convention (default)
[git]
branch_prefix = "feature/"

# Explicit overrides when naming doesn't match
[git.branch_specs]
"feature/redesign-2026" = "spec-047"
"hotfix/auth-regression" = "spec-023"
```

### Worktree Support (alpha: basic)

Alpha ships worktree awareness — Shipwright knows when it's running inside a worktree, registers it in SQLite, and generates correctly scoped context. Full worktree orchestration (agent spawning, parallel coordination) is V2.

```
ship worktree create <spec-id>   # create worktree for spec, generate context
ship worktree list               # list active worktrees + status
ship worktree archive <id>       # merge done, clean up
```

## Configurable Workflows

Teams define their own loop. The default is sensible. Override what doesn't fit.

```toml
# .ship/config.toml

[workflow]
preset = "solo"    # "solo" | "team" | "custom"
```

**solo** — Note → Spec → Issues → Kanban. No features, no worktrees (available but not surfaced).

**team** — Note → Spec → Feature branch → Issues → Kanban → Worktrees. Full git integration.

**custom:**

```toml
[workflow]
preset = "custom"
document_types = ["spec", "issue"]     # no notes, no adrs

[workflow.transitions]
note_to_spec = false
spec_to_feature = false                # straight to issues
spec_to_issues = true
feature_to_worktree = false
```

UI responds to workflow config — disabled document types don't appear in nav.

## Modes (Alpha: Basic)

Modes shape the Shipwright UI and MCP tool surface for the kind of work being done.

**Modes are about human intent. Branch config is about agent environment. They are different things.**

- Modes: manually switched, global to Shipwright session, control UI layout and Shipwright MCP tool surface
- Branch config: automatic on checkout, control external MCP servers, model, skills, context

Alpha ships two default modes (Planning, Execution) with manual switching. Mode-responsive UI complexity is V1 — alpha just switches the active mode label and MCP tool filter.

```toml
# .ship/modes/planning.toml
id = "planning"
name = "Planning"
shipwright_tools = ["list_specs", "create_spec", "list_issues", "create_issue", "list_notes"]
color = "#6366f1"

# .ship/modes/execution.toml
id = "execution"
name = "Execution"
shipwright_tools = ["list_issues", "get_issue", "move_issue", "update_issue", "link_issues"]
color = "#10b981"
```

## Desktop App (Tauri)

React + TypeScript + Tailwind CSS v4. Tauri 2.x. Specta for type generation.

### Views

**Notes** — Chronological list. Simple editor. "Promote to Spec" button. Deliberately lightweight.

**Kanban** — Default landing view. Columns from config statuses. Cards show title, assignee, tags. Drag-and-drop moves the file to the correct status folder. Click to open issue detail. Hidden statuses (archived) don't appear as columns.

**Issue Detail** — Full markdown render. Edit in place. Frontmatter fields as a form. Links to related issues and parent spec.

**Spec Editor** — Split view. Left: editable markdown. Right: AI conversation scoped to this spec via MCP sampling. "Extract Issue" button creates issue pre-populated from spec context.

**ADR List** — Table: status, date, title. Click to read. Button to create.

**Settings** — Three tabs: Project, Global, Agents.

- Project: name, statuses, tags, git behavior, workflow preset, templates
- Global: user info, editor, MCP port, theme
- Agents: active mode selector, managed AI tools (alpha: export buttons only), branch context viewer (what's in the current CLAUDE.md)

### Quality Bar

This is a commercial product. Every interaction on the happy path must be smooth. Typography and spacing must be consistent. It should feel like Linear — not a developer's weekend project. Empty states are first impressions: one sentence explaining what the thing is, one CTA.

## MCP Server

The alpha's primary value proposition. Persistent project memory across agent sessions.

### What This Enables

An agent starts a new session. It reads Shipwright's MCP tools — open issues, current spec, prior ADRs, action log. It does its work, updates issue status, files a decision as an ADR. The next session starts with full context. No re-explaining. No lost memory.

CLAUDE.md on the current branch tells the agent how to use Shipwright's tools. The agent is Shipwright-native from the first message.

### Alpha MCP Tools

```
# Notes
ship_list_notes
ship_get_note
ship_create_note

# Specs
ship_list_specs
ship_get_spec
ship_create_spec
ship_update_spec

# Issues
ship_list_issues
ship_get_issue
ship_create_issue
ship_update_issue
ship_move_issue
ship_link_issues

# ADRs
ship_list_adrs
ship_get_adr
ship_create_adr

# Project context
ship_get_project_info
ship_get_log
ship_get_branch_context    # current branch, linked spec, active skills

# Sampling (user-initiated only)
ship_refine_spec
ship_extract_issues
ship_draft_adr
```

### MCP Sampling

Shipwright uses MCP sampling — requests completions from the user's connected AI client, not from any API it controls. No API keys in Shipwright. No model costs charged through Shipwright. Works with Claude, Cursor, Windsurf, or any MCP-compatible client.

Sampling in alpha is always user-initiated. No auto-triggers.

### Action Log (`log.md`)

Every write operation appends to `.ship/log.md`:

```
2026-02-22T14:30:00Z [human]        issue-001 moved in-progress → review
2026-02-22T14:35:00Z [agent:claude] issue-001 updated: added task breakdown
2026-02-22T14:36:00Z [agent:claude] adr-003 created: "Use Redis for session storage"
2026-02-22T14:38:00Z [hook]         post-checkout: generated CLAUDE.md for feature/auth
2026-02-22T14:42:00Z [human]        mode changed: planning → execution
```

Append-only. Human-readable. Tracked in git by default (configurable).

## Resolved Decisions

These were open questions in v0.1 — now locked:

**Issue IDs:** `issue-NNN` format. Auto-incrementing, human-readable, CLI-friendly.

**MCP server lifecycle:** Auto-starts when `ship ui` launches. Visible status indicator in the UI. Explicit `ship mcp start` also available for headless use.

**Spec conversation persistence:** Ephemeral for alpha — log.md records that a conversation happened, not the full transcript. Revisit in V1.

**`ship init` + git:** Detects existing git repo, installs hooks, appends generated file paths to `.gitignore`, offers to commit `.ship/` as initial commit.

**`.ship` vs `.shipwright`:** `.ship` — shorter, terminal-friendly, already in the codebase.

**`logic/` → `runtime/`:** Yes — rename now at 15K LOC, not later at 90K. Better name, correct architecture signal.

**`plugins/` → `modules/`:** Yes — modules are compiled-in first-party code, not plugins. Extensions (third-party, sandboxed) come in V2.

**Generated files policy:** CLAUDE.md, .mcp.json, .gemini/, .codex/, .claude/ are never committed in a Shipwright project. `ship init` writes these to `.gitignore`. `pre-commit` hook enforces it.

## Alpha Done Criteria

Alpha is done when the core loop works end-to-end without friction:

1. `ship init` in a new project completes in under 10 seconds, git hooks installed, `.gitignore` updated
2. `ship note "auth redesign thoughts"` creates a timestamped note instantly
3. Note promoted to spec — single action, clean redirect
4. Spec refined in split view with AI conversation
5. Issues extracted from spec via "Extract Issues" button or MCP sampling
6. Issues appear on Kanban board
7. Drag-and-drop moves an issue — file moves, timestamp updates, no flicker
8. `git checkout feature/auth` fires `post-checkout`, CLAUDE.md appears in project root
9. Open Claude Code, connect to Shipwright MCP server (`ship mcp start --stdio`)
10. Agent reads spec and open issues without being told they exist — CLAUDE.md guided it
11. Agent updates an issue — change appears in Kanban within 1 second
12. `log.md` shows a coherent history of human, agent, and hook actions
13. Staging CLAUDE.md or .mcp.json is blocked by `pre-commit` hook
14. `ship worktree create spec-001` creates a worktree, context generated inside it
15. `ship tools export --target claude` writes correct `.mcp.json` from branch config
16. No account. No internet. One binary.
