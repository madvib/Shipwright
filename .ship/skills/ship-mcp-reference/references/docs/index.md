# MCP Server Overview

The Ship MCP server (`apps/mcp/`) exposes runtime operations to AI agents via the Model Context Protocol. It is the primary programmatic interface to Ship's workspace, session, job, and project state.

## Two Server Variants

### ShipServer (Full)

The primary server. Runs via `ship mcp serve` (stdio) or `ship mcp serve --http --port <N>` (HTTP). Features:

- All stable tools (project, agent, workspace, session, skills, events, studio sync)
- All unstable tools when compiled with `--features unstable` (notes, ADRs, targets, capabilities, jobs, file claims)
- Resource support (`ship://` URIs) for read-only access to project state
- Tool gating based on the active agent profile's `active_tools` list
- Resource change notifications to connected clients

### StudioServer (Subset)

Runs via `ship studio --port <N>`. Designed for the Ship Studio web IDE. Features:

- Project, studio sync, skills, and vars tools only
- No tool gating -- all registered tools are always callable
- No resource support
- Listens on HTTP for browser connections

## Starting the Server

The standard MCP server is configured in `.mcp.json`, generated by `ship use`:

```json
{
  "mcpServers": {
    "ship": { "command": "ship", "args": ["mcp", "serve"] }
  }
}
```

For HTTP transport:

```
ship mcp serve --http --port 8080
```

For the Studio server:

```
ship studio --port 3001
```

The Studio server prints the MCP endpoint URL and optionally opens the Studio web app.

## Project Resolution

On first use, call `open_project` with the absolute path to the project root. This sets the active project for all subsequent tool calls.

If a project is not explicitly opened, the server attempts to resolve it from the current working directory by walking up the tree looking for `.ship/`.

## Tool Gating

The full server gates tool visibility based on the active agent profile.

**Core tools** are always visible regardless of agent configuration. These include: `open_project`, `activate_workspace`, `create_workspace`, `complete_workspace`, `list_stale_worktrees`, `list_workspaces`, `set_agent`, `start_session`, `end_session`, `log_progress`, `list_skills`, `list_events`, `push_bundle`, `pull_agents`, `list_local_agents`, `write_skill_file`, `delete_skill_file`, `list_project_skills`.

**Unstable tools** (compiled with the `unstable` feature flag) are also core: `create_note`, `update_note`, `create_adr`, `create_job`, `update_job`, `list_jobs`, `append_job_log`, `claim_file`, `get_file_owner`, `create_target`, `update_target`, `list_targets`, `get_target`, `create_capability`, `update_capability`, `delete_capability`, `mark_capability_actual`, `list_capabilities`.

When an agent profile defines an `active_tools` list, only core tools and tools in that list are visible. If `active_tools` is empty, all tools are visible.

Tool names are normalized before matching: hyphens become underscores, `ship_` prefix and `_tool` suffix are stripped, and the name is lowercased.

{% aside type="tip" %}
The Studio server has no tool gate. All registered tools are always callable, making it suitable for interactive IDE use.
{% /aside %}

## Branch Resolution

Session tools (`start_session`, `end_session`, `log_progress`) accept an optional `branch` parameter. If omitted, the server resolves the current git branch by running `git branch --show-current` in the project root directory.
