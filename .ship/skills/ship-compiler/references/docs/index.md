# Ship Compiler

Ship is a compiler for AI agent configuration. It takes a declarative `.ship/` directory and emits provider-specific config files (CLAUDE.md, .mcp.json, .cursor/rules/, etc.). Those outputs are build artifacts -- gitignored, ephemeral, regenerated by `ship use`.

## What the Compiler Does

The compiler transforms a single agent definition into config files for every supported provider. Write one `.ship/agents/my-agent.jsonc`, run `ship use my-agent`, and get working config for Claude Code, Gemini CLI, Codex, Cursor, and OpenCode simultaneously.

The compiler is a pure function. It takes structured data as input and returns strings as output. No filesystem access, no network calls, no database queries. This purity makes it safe to run in any context.

## Three Stages

The compilation pipeline has three stages:

```
.ship/ directory
       |
       v
  ProjectLibrary  (load)
       |
       v
  ResolvedConfig  (resolve)
       |
       v
  CompileOutput   (compile)
       |
       v
  Provider-native files on disk
```

**Stage 1: Load** reads the `.ship/` directory into a `ProjectLibrary` struct. This includes agent profiles from `agents/*.jsonc`, skill definitions from `skills/*/SKILL.md`, MCP server declarations from `mcp.jsonc`, permission presets from `permissions.jsonc`, rules from `rules/`, and the project manifest `ship.jsonc`. After loading, no further filesystem access occurs.

**Stage 2: Resolve** takes the `ProjectLibrary` plus the active agent ID and produces a `ResolvedConfig`. This stage merges project defaults with the agent profile and any workspace overrides. All skill references are expanded, all MCP server IDs are resolved to full configurations, and all permission presets are flattened. The output is a self-contained configuration with no unresolved references.

**Stage 3: Compile** takes the `ResolvedConfig` and a target provider ID. It emits a `CompileOutput` containing every file the provider needs: context content, MCP server entries, skill files mapped to provider-native paths, settings patches, rule files, agent definitions, and plugin manifests.

## WASM Compilation Target

The compiler crate compiles to both native (for the CLI and MCP server) and WASM (for Ship Studio in the browser). The WASM entry point is `compile_library()`, which accepts a JSON-serialized `ProjectLibrary` and returns compiled output.

This architecture guarantees:

- The CLI, MCP server, and web UI all use identical compilation logic.
- No server round-trip is needed for compilation in the browser.
- The `@ship/compiler` npm package wraps the WASM output for frontend consumption.

## Key CLI Commands

| Command | Effect |
|---------|--------|
| `ship use <agent-id>` | Load, resolve, compile, and write for all agent providers |
| `ship compile` | Recompile the active agent |
| `ship compile --dry-run` | Preview what would be written without touching files |
| `ship compile --provider <id>` | Compile for a single provider only |
| `ship validate` | Check `.ship/` config for errors before compile |
| `ship status` | Show active agent and compilation status |

## Design Constraints

**Compiler is pure.** No filesystem, no network, no database. `ProjectLibrary` in, `CompileOutput` out. This makes it safe to run in WASM and straightforward to test.

**Idempotent by default.** `ship use` can be run repeatedly. The compiler overwrites artifacts. Running it twice produces identical output.

**All outputs are gitignored.** Compiled files are build artifacts. The `.ship/` directory is source; everything the compiler writes is derived.
