Ship compiles a single agent profile into provider-native config files. Each provider has its own output format. All output files are gitignored build artifacts, regenerated by `ship use` or `ship compile`.

## Supported Providers

| Provider | ID | Context format | MCP format | Skills format |
|----------|-----|---------------|------------|--------------|
| Claude Code | `claude` | Markdown | JSON | Markdown in directory |
| Gemini | `gemini` | Markdown | JSON | Markdown in directory |
| Codex | `codex` | Markdown | TOML | Markdown in directory |
| Cursor | `cursor` | MDC rule files | JSON | Markdown in directory |

## Claude (`claude`)

The default provider. Outputs:

| File | Purpose |
|------|---------|
| `CLAUDE.md` | Agent context -- rules, inline instructions, skill content |
| `.mcp.json` | MCP server definitions for Claude Code |
| `.claude/skills/<name>/SKILL.md` | Individual skill files in Claude's native format |

`CLAUDE.md` is a single markdown file containing all compiled agent instructions, rules, and resolved skill content. Claude Code reads it automatically from the project root.

`.mcp.json` follows the Claude Code MCP configuration format with server entries containing `command`, `args`, and `env` fields.

### Example `.mcp.json`

```json
{
  "mcpServers": {
    "ship": {
      "command": "ship",
      "args": ["mcp", "serve"],
      "env": {}
    }
  }
}
```

## Gemini (`gemini`)

| File | Purpose |
|------|---------|
| `GEMINI.md` | Agent context -- rules, inline instructions, skill content |
| `.gemini/settings.json` | MCP server configuration and workspace policy |
| `.gemini/skills/<name>/SKILL.md` | Individual skill files |

`GEMINI.md` serves the same role as `CLAUDE.md` but formatted for Gemini's context window. The `.gemini/` directory contains both the settings file and a workspace policy.

## Codex (`codex`)

| File | Purpose |
|------|---------|
| `AGENTS.md` | Agent context -- rules, inline instructions, skill content |
| `.codex/config.toml` | MCP server configuration in TOML format |
| `.agents/skills/<name>/SKILL.md` | Individual skill files |

Codex uses `AGENTS.md` as its primary context file and TOML for MCP configuration.

## Cursor (`cursor`)

| File | Purpose |
|------|---------|
| `.cursor/rules/<name>.mdc` | Rule files with YAML frontmatter |
| `.cursor/mcp.json` | MCP server definitions |
| `.cursor/skills/<name>/SKILL.md` | Individual skill files |
| `.cursor/cli.json` | Permissions and tool configuration |
| `.cursor/hooks.json` | Pre/post tool hooks (when configured) |
| `.cursor/environment.json` | Environment variables for MCP servers |

Cursor does not use a single context markdown file. Instead, agent instructions are split into `.mdc` rule files under `.cursor/rules/`. Each rule file has YAML frontmatter controlling when the rule applies.

### Example `.cursor/rules/style.mdc`

```markdown
---
description: Code style conventions
globs: ["**/*.ts", "**/*.tsx"]
---

Follow existing code conventions. Use explicit types.
```

## Multi-Provider Compilation

Target multiple providers from a single agent profile:

```jsonc
{
  "agent": {
    "id": "my-agent",
    "providers": ["claude", "gemini", "codex", "cursor"]
  }
}
```

`ship use my-agent` compiles for every listed provider simultaneously. To compile for a single provider:

```bash
ship compile --provider claude
ship compile --provider gemini
```

## Relationship to `.ship/`

The `.ship/` directory is source configuration -- it is committed to version control. The compiled output files are build artifacts that should never be committed. The `.ship/.gitignore` file ensures this.

```
# Source (committed)
.ship/agents/my-agent.jsonc
.ship/skills/code-review/SKILL.md
.ship/mcp.jsonc

# Compiled output (gitignored)
CLAUDE.md
.mcp.json
.claude/skills/
```

Run `ship compile --dry-run` to preview what would be written without touching the filesystem.
