The Ship MCP server exposes platform tools to AI agents via the Model Context Protocol. It is the primary interface between agents and Ship's runtime -- managing workspaces, sessions, jobs, targets, and project state.

## Starting the Server

The MCP server is configured in `.mcp.json`, generated by `ship use`:

```json
{
  "mcpServers": {
    "ship": { "command": "ship", "args": ["mcp", "serve"] }
  }
}
```

On first use in a new directory, register the project:

```
open_project(path="/absolute/path/to/project")
```

## Why MCP Tools Instead of Direct Access

Ship stores state in a SQLite database (`platform.db`). Never query it directly. The MCP tools and CLI enforce schema invariants, emit events to the append-only log, and handle migrations. Direct database access bypasses all of these guarantees.

## The Three-Stage Workflow

Every agent interaction follows a three-stage pattern.

### Stage 1: Planning

Read project state to understand what exists and what needs doing.

```
open_project(path="/path/to/project")
```

Then read resources for context:
- `ship://project_info` -- project metadata and active configuration
- `ship://targets` -- milestones and surface targets
- `ship://jobs` -- current job queue
- `ship://workspaces` -- active workspaces

Use `create_note` or `create_adr` to record planning decisions.

### Stage 2: Workspace

Create or activate a workspace for the unit of work.

```
create_workspace(name="auth-flow", kind="imperative", base_branch="main")
activate_workspace(branch="auth-flow")
set_agent(id="web-specialist")
```

Workspace kinds:
- **imperative** -- creates a git worktree, pruned on completion
- **declarative** -- creates a git worktree, kept on completion
- **service** -- no worktree, for standing processes

### Stage 3: Session

Start a session, do the work, log progress, end with a summary.

```
start_session(goal="Implement GitHub OAuth login")
log_progress(note="Created auth provider config")
log_progress(note="Login UI renders, testing OAuth flow")
end_session(summary="GitHub OAuth flow working end-to-end")
```

## Tool Availability

Tools are organized into two tiers:

**Core tools** are always available: `open_project`, `create_note`, `update_note`, `create_adr`, workspace tools, session tools, job tools, `list_skills`, `list_events`.

**Extended tools** require an agent mode that includes them in its `active_tools` list: `create_target`, `list_targets`, `get_target`, `create_capability`, `mark_capability_actual`, `list_capabilities`. Call `set_agent` to activate a mode that includes the tools you need.

## Resources vs Tools

Resources (`ship://` URIs) are read-only and return bulk data. Use them when you need to read project state without side effects. Tools perform mutations and return structured responses.

For read-heavy workflows (understanding project state, reviewing what exists), prefer resources. For mutations (creating workspaces, logging progress, updating jobs), use tools.
