# Ship MCP Tools Reference

The Ship MCP server (`ship mcp serve`) exposes platform tools to agents at runtime via the Model Context Protocol. Agents call these tools to manage workspaces, sessions, jobs, and project state.

**Never access the database directly** (`sqlite3 ~/.ship/platform.db`). Always use these tools or the `ship` CLI — they enforce schema invariants, emit events, and handle migrations.

---

## Getting started

The MCP server is configured in `.mcp.json` (generated by `ship use`):

```json
{
  "mcpServers": {
    "ship": { "command": "ship", "args": ["mcp", "serve"] }
  }
}
```

On first use in a new directory, call `open_project` to register it:

```
open_project(path="/absolute/path/to/project")
```

---

## Core Tools

These tools are always available, regardless of which agent mode is active.

---

### Project

#### `open_project`

Register the active project with the MCP server. Call once per session when the server starts in a new directory.

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `path` | string | yes | Absolute path to the project root |

### Notes & ADRs

#### `create_note`

Create a project note in the Ship database. Notes are cross-session records — use them for decisions, blockers, or cross-lane signals.

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `title` | string | yes | Note title |
| `content` | string | no | Markdown content |
| `branch` | string | no | Git branch to associate with this note |

#### `update_note`

Replace the content of an existing note.

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | string | yes | Note ID (nanoid returned by `create_note`) |
| `content` | string | yes | Full replacement markdown content |
| `scope` | string | no | `project` (default) or `user` |

#### `create_adr`

Record an Architecture Decision Record. Use for decisions with real alternatives that constrain future work.

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `title` | string | yes | Short, specific title — verb + noun (e.g. "Use D1 for cloud state") |
| `decision` | string | yes | Full decision text: context, decision, alternatives, consequences |

See the `write-adr` skill for the recommended structure.

### Workspaces

Workspaces are the primary unit of parallel work. Each workspace corresponds to a git worktree (for imperative/declarative kinds) or a standing service branch.

#### `activate_workspace`

Mark a workspace as active for the current session.

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `branch` | string | yes | Workspace branch/id |
| `agent_id` | string | no | Agent profile to associate with this workspace |

#### `create_workspace`

Create a new workspace and its git worktree (for non-service kinds).

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | string | yes | Human-readable workspace name |
| `kind` | string | yes | `imperative` \| `declarative` \| `service` |
| `preset_id` | string | no | Agent profile to activate in this workspace |
| `branch` | string | no | Branch name (derived from name if omitted) |
| `base_branch` | string | no | Base branch for worktree (default: `main`) |
| `file_scope` | string | no | Paths this workspace may edit (e.g. `crates/`) |

#### `complete_workspace`

Mark a workspace complete and write a handoff summary.

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `workspace_id` | string | yes | Workspace branch/id |
| `summary` | string | yes | What was accomplished — written to handoff.md |
| `prune_worktree` | bool | no | Remove the git worktree (default: true for imperative) |

#### `list_workspaces`

List all workspaces for the active project.

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `status` | string | no | Filter: `active`, `idle`, `archived` |

#### `list_stale_worktrees`

Find worktrees idle beyond a threshold. Use to identify candidates for cleanup.

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `idle_hours` | integer | no | Threshold in hours (default: 24) |

#### `set_agent`

Set the active agent profile for the current workspace context.

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | string | no | Agent profile ID. Omit to clear. |

### Sessions

Sessions track agent activity within a workspace. One session per agent visit.

#### `start_session`

Begin a session for the current workspace. Call at the start of each agent visit.

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `branch` | string | no | Workspace branch (resolves from git if omitted) |
| `goal` | string | no | Session goal — what this visit aims to accomplish |
| `agent_id` | string | no | Agent profile override |
| `provider_id` | string | no | Provider override (e.g. `claude`, `codex`) |

#### `end_session`

End the current session. Write a summary of what was accomplished.

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `branch` | string | no | Workspace branch (resolves from git if omitted) |
| `summary` | string | no | What was accomplished, what changed |

#### `log_progress`

Record a progress checkpoint within the current session. Call at each meaningful milestone.

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `note` | string | yes | What you did, decided, or got blocked on |
| `branch` | string | no | Workspace branch (resolves from git if omitted) |

Requires an active session (call `start_session` first).

### Jobs

Jobs are units of coordinated work. The job queue is the coordination layer across agents and machines.

#### `create_job`

Create a new job in the queue.

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `kind` | string | yes | Job kind: `feature`, `fix`, `infra`, `test`, `review`, etc. |
| `description` | string | yes | What needs to be done |
| `branch` | string | no | Git branch this job is associated with |
| `assigned_to` | string | no | Agent id or workspace to assign to |
| `requesting_workspace` | string | no | Workspace that requested this job |
| `priority` | integer | no | Scheduling priority — higher runs first (default 0) |
| `blocked_by` | string | no | Job id that must complete first |
| `touched_files` | string[] | no | Files this job intends to touch |
| `file_scope` | string[] | no | File/dir paths the agent may touch — enforced by gate |
| `acceptance_criteria` | string[] | no | Checklist items for the gate |
| `capability_id` | string | no | Capability this job delivers |
| `preset_hint` | string | no | Profile to compile in the worktree |
| `symlink_name` | string | no | Human-readable worktree label |

#### `update_job`

Update a job's status, assignment, or priority.

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | string | yes | Job id (or unique prefix) |
| `status` | string | no | `pending` \| `running` \| `complete` \| `failed` |
| `assigned_to` | string | no | Reassign to a different agent |
| `priority` | integer | no | Update scheduling priority |
| `blocked_by` | string | no | Set or clear blocking job id |
| `touched_files` | string[] | no | Replace the touched files list |

#### `list_jobs`

List jobs, optionally filtered.

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `status` | string | no | Filter: `pending`, `running`, `complete`, `failed` |
| `branch` | string | no | Filter by branch |

#### `append_job_log`

Append a log message to a job. Use to record progress, warnings, or touched files.

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `job_id` | string | yes | Job id |
| `message` | string | yes | Log message. Prefix with `touched: path/to/file` to register file ownership. |
| `level` | string | no | `info`, `warn`, or `error` |

**File ownership shorthand:** `append_job_log(job_id, "touched: apps/web/src/foo.ts")` automatically registers that file as owned by this job.

#### `claim_file`

Claim exclusive ownership of a file for a job. Returns a conflict error if already owned by another running job.

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `job_id` | string | yes | Job claiming ownership |
| `path` | string | yes | File path relative to project root |

#### `get_file_owner`

Look up which job owns a file.

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `path` | string | yes | File path relative to project root |

### Targets & Capabilities

Targets are milestones or product surfaces. Capabilities are the specific features/behaviors that belong to a target. Together they form the north-star tracking layer.

#### `create_target`

Create a milestone or surface target.

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `kind` | string | yes | `milestone` (e.g. v0.1.0) or `surface` (e.g. compiler) |
| `title` | string | yes | Short title |
| `description` | string | no | Longer description |
| `goal` | string | no | One-line north star goal |
| `status` | string | no | `active`, `planned`, `complete`, `frozen` (default: `active`) |

#### `list_targets`

List all targets.

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `kind` | string | no | Filter: `milestone` or `surface` |

#### `get_target`

Get a target and its capabilities (actual vs aspirational).

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | string | yes | Target id |

Returns the target with all associated capabilities grouped by status.

#### `create_capability`

Add a capability to a target.

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `target_id` | string | yes | Target this capability belongs to |
| `title` | string | yes | Capability title |
| `milestone_id` | string | no | Milestone this capability is required for |

#### `mark_capability_actual`

Mark a capability as delivered. Requires concrete evidence.

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | string | yes | Capability id |
| `evidence` | string | yes | Proof: test name, commit hash, or observable behavior |

Never mark a capability actual without evidence. The gate protocol exists to enforce this.

#### `list_capabilities`

List capabilities, filtered by target or milestone.

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `target_id` | string | no | Filter by surface target id |
| `milestone_id` | string | no | Filter by milestone (across all surfaces) |
| `status` | string | no | `aspirational` or `actual` |

---

### Events

#### `list_events`

Query the project event log. Events are append-only and cannot be modified.

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `since` | string | no | ISO 8601 or relative: `1h`, `24h`, `7d` |
| `actor` | string | no | Filter by actor (substring) |
| `entity` | string | no | Entity type: `workspace`, `session`, `note`, `adr`, etc. |
| `action` | string | no | Action: `create`, `update`, `delete`, `start`, `stop`, etc. |
| `limit` | integer | no | Max results (default 50, max 200) |

---

### Compiler

#### `provider_matrix`

Show which providers are supported and what config files each generates.

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `provider` | string | no | Filter to one provider: `claude`, `gemini`, `codex`, `cursor` |
| `format` | string | no | `json` (default), `text`, or `diff` |

## Extended Tools

Some tools are only available when a specific mode is active (defined in the mode's `active_tools` list). Call `set_agent` to activate a mode that includes the tools you need.

Extended tools include: `create_target`, `list_targets`, `get_target`, `create_capability`, `mark_capability_actual`, `list_capabilities` — these require an agent mode that includes them.

---

## Tool availability by mode

Tools are gated by the active agent mode's `active_tools` list. Core tools (listed above) bypass this gate and are always available. Extended tools require explicit activation.

To check which tools are available: `list_skills` returns the active configuration, and the mode's profile TOML shows `active_tools`.
